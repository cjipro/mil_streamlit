Section 1: Three-Layer Operating Model
Manifest is canonical. Brief and policy set boundaries. Jira and GitLab are views for human convenience. No tool other than the manifest is source of truth for what has been built, what remains to be built, or what an AI agent should do next.



Section 2: Dual Closure Rule — No Exceptions
Every ticket requires two independent closure conditions. Neither alone is sufficient. This rule applies to every ticket in this plan without exception.


A ticket is not closed until both conditions are met. Moving a Jira ticket to Done without both conditions is a governance violation.

Section 3: Manifest Schema — Every Component, Every Field
Every component in the system has a manifest file. The manifest is what an AI agent reads to understand, build, and rebuild that component. This section defines the mandatory schema. Every field is required unless marked optional.

3.1 Core Component Fields (all manifests)

3.2 v5 Addition: recovery_patterns (all manifests)
Every component manifest must include recovery_patterns. This is what an AI agent does when a component fails. Without this field, failure produces ambiguity. With it, failure produces a clear path.

3.3 v5 Addition: interfaces (all manifests)
Interfaces define the contract between components — what this component expects from its inputs and promises to its outputs. This is what prevents integration failures.

3.4 v5 Addition: preferred_model (agent manifests)
All agent manifests must declare their preferred model and fallback. Model names must never be hardcoded in agent code — always read from config/model_config.yaml which is in turn referenced in the manifest.

3.5 v5 Addition: graduated_trust_tier + runtime_permissions (all manifests)

3.6 v5 Addition: governance_attributes (data contract manifests only)

3.7 v5 Addition: prompt_breakdown (complex ticket manifests)
Complex tickets (estimated >4h) require a machine-readable sequential breakdown in the manifest. This is the structure that just-in-time prompts are written against. The breakdown is the map. The prompts are the directions.


Section 4: Open Audit Findings — Must Resolve Before Pipeline Build
These findings were identified before any pipeline code was written. Finding 002 is CRITICAL. It blocks the entire Phase 1 mission. No pipeline code may be written before audit findings are formally resolved or accepted with documented fallback.


DATA OWNER SIGN-OFF REQUIRED: All three findings require data owner sign-off before marking RESOLVED or ACCEPTED-WITH-FALLBACK. This is a non-negotiable governance requirement added in v5 (TDWI recommendation). No finding may be marked resolved without named data owner signature and date.

Section 7: Governance, Decision Rights & Controlled Agent Path
7.1 Graduated Trust Model — Tiers 1–4
Governance is a ramp, not a wall. Journey owners earn expanded permissions through demonstrated competence. The path from read-only to controlled deployer is defined, achievable, and governed.


7.2 Controlled Agent Path
Agents in CJI Pulse are intelligence agents, not action agents. They interpret, classify, and recommend. They do not act. This section defines what agents may and may not do.
