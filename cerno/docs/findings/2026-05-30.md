# Findings — 2026-05-30

## Context

While building an exact-replica dev environment on Windows (conda env at
`C:/Users/hussa/miniconda3/envs/cerno-3.11`, Python 3.11.15 + Java 8
Zulu OpenJDK + all approved libs at bank-edge pinned versions), an
upstream incompatibility surfaced between `pyspark==2.4` and Python 3.11.

## Observations

- `pyspark==2.4` installs cleanly from conda-forge alongside Python 3.11
  + openjdk=8. Resolver is happy.
- `import pyspark` fails with `TypeError: code expected at least 16
  arguments, got 15`. The traceback ends in
  `pyspark/cloudpickle.py::_make_cell_set_template_code` calling
  `types.CodeType(...)` with the pre-3.11 signature.
- Python 3.11 (PEP 657 / 3.11 release notes) extended `types.CodeType`
  to require `qualname` and an exception-handling table. PySpark 2.4.x's
  vendored cloudpickle was last touched circa 2018 and never received
  the patch.
- The 45/45 cerno test suite is unaffected because `cerno.spark` uses a
  lazy `from pyspark.sql import SparkSession` inside `get_spark` —
  nothing imports pyspark at module-load time.
- Java side is fine: Zulu OpenJDK 1.8.0_472 in-env, plus a separate
  Temurin 1.8.0_492 extracted at `cerno/.jdk-8/`. Both run cleanly.
- All other libs landed at exact bank-edge pins: duckdb 1.5.2,
  pyarrow 18.1.0, numpy 1.26.4, pandas 2.2.3, scipy 1.13.1,
  PyYAML 6.0.1, joblib 1.4.2, scikit-learn 1.5.1, statsmodels 0.14.2,
  pyodbc 5.1.0.
